1. Resumen Ejecutivo

El presente documento tiene como objetivo informar al cliente sobre el estado actual del desarrollo del sistema PremiumBus, una aplicación móvil web progresiva (PWA) diseñada para la gestión integral de compra de boletos de transporte urbano en la ciudad de San Luis Potosí, S.L.P.

A la fecha de elaboración de este informe (13 de abril de 2026), el sistema se encuentra en una fase avanzada de desarrollo, con la mayor parte de los módulos funcionales completados y operativos. El avance general estimado del proyecto es del 85%.

85% Completado
💡 Punto clave: La aplicación ya es funcional en modo demostración (offline con localStorage) y cuenta con una API REST completa lista para conectarse a un servidor MySQL de producción. Se han integrado 30 rutas reales de transporte urbano de San Luis Potosí con datos georeferenciados.

5. Módulo Backend (API REST / Base de Datos)

5.1 API REST — PHP

Se desarrolló una API REST completa en PHP que comunica la aplicación frontend con la base de datos MySQL. La API implementa los siguientes 10 endpoints:

Método Endpoint Descripción Estado
POST ?action=register Registrar nuevo usuario ✅
POST ?action=login Autenticar usuario ✅
GET ?action=trips Obtener todos los viajes activos ✅
GET ?action=trip&id=X Obtener viaje específico por ID ✅
POST ?action=purchase Registrar compra de boleto ✅
GET ?action=user_purchases Compras de un usuario específico ✅
GET ?action=all_purchases Todas las compras (panel admin) ✅
GET ?action=all_users Lista de usuarios registrados (admin) ✅
GET ?action=occupied_seats Asientos ocupados por viaje ✅
GET ?action=health Verificación de salud del servidor ✅

5.2 Medidas de Seguridad Implementadas

  • Prepared Statements (PDO): Prevención de inyección SQL en todas las consultas
  • Hashing de contraseñas: Encriptación SHA-256 para almacenamiento seguro
  • Validación de entrada: Verificación de tipos, formatos y longitudes en servidor
  • Headers CORS: Control de acceso entre orígenes configurado
  • Unicidad de asientos: Restricción UNIQUE en base de datos para evitar doble reserva

5.3 Base de Datos MySQL

El esquema de base de datos contempla 3 tablas principales con relaciones e índices optimizados:

Tabla Campos Principales Relaciones
usuarios id, nombre, correo (UNIQUE), password, rol, created_at PK: id — Referenciada por compras
viajes id, nombre_ruta, origen, destino, coordenadas (lat/lng), paradas (JSON), fecha_salida, precio, asientos_totales, activo PK: id — Referenciada por compras
compras id, usuario_id (FK), viaje_id (FK), nombre_ruta, origen, destino, fecha, asiento, precio, fecha_compra FK: usuario_id → usuarios.id, viaje_id → viajes.id

5.4 Estrategia Dual (Online / Offline)

Arquitectura innovadora: El sistema implementa una estrategia de backend dual:
  • Modo Online: Si el servidor PHP/MySQL está disponible, la aplicación se conecta automáticamente a la API REST.
  • Modo Demo (Offline): Si el servidor no está disponible, la aplicación funciona completamente usando localStorage como capa de datos, con datos seed de 30 rutas pregargadas.
Esta estrategia permite demostrar la aplicación al cliente sin necesidad de un servidor activo.

9. Riesgos y Observaciones

ID Riesgo / Observación Impacto Mitigación
R-01 El servidor de producción (hosting PHP/MySQL) aún no ha sido configurado Alto Se requiere contratar un servicio de hosting que soporte PHP 7.4+ y MySQL 5.7+. La aplicación funciona en modo demo mientras tanto.
R-02 SHA-256 es menos seguro que bcrypt para hashing de contraseñas Medio Se recomienda migrar a password_hash() de PHP con algoritmo BCRYPT antes del despliegue a producción.
R-03 No se ha implementado HTTPS en el entorno de desarrollo Medio El hosting de producción debe incluir certificado SSL/TLS. PWA requiere HTTPS para funcionamiento completo.
R-04 El panel administrativo tiene funcionalidad limitada Bajo Las funciones de edición y eliminación de registros se agregarán en el siguiente sprint.
R-05 No hay sistema de pagos integrado (pasarela de pagos) Medio Se sugiere evaluar la integración con plataformas como Mercado Pago, OpenPay o Stripe para procesamiento real de pagos.
R-06 Las pruebas de integración con servidor real están pendientes Medio Una vez configurado el hosting, se ejecutarán pruebas end-to-end de todos los flujos.

12. Conclusiones y Recomendaciones

12.1 Conclusiones

El desarrollo del sistema PremiumBus se encuentra en un estado avanzado con un 85% de avance general. Los módulos fundamentales del sistema — autenticación, consulta de viajes, visualización de mapas y compra de boletos — están completamente funcionales y han sido probados satisfactoriamente en modo demostración.

La arquitectura implementada es sólida, modular y escalable, siguiendo principios de separación de responsabilidades (SoC) y permitiendo fácil mantenimiento futuro. La capacidad de funcionar tanto con backend MySQL como con localStorage (modo demo) demuestra la robustez del diseño.

Se han integrado 30 rutas reales de transporte de San Luis Potosí con datos georeferenciados reales, lo cual aporta valor significativo a la experiencia del usuario.

12.2 Recomendaciones

  1. Priorizar la contratación del hosting para proceder con el despliegue a producción lo antes posible.
  2. Realizar pruebas de aceptación de usuario (UAT) una vez desplegado el sistema para validar los flujos completos.
  3. Evaluar la integración de una pasarela de pagos si se requiere procesamiento real de transacciones monetarias.
  4. Mejorar la seguridad de contraseñas migrando de SHA-256 a bcrypt antes de la salida a producción.
  5. Considerar una fase 2 del proyecto para funcionalidades avanzadas como: notificaciones push, historial detallado de viajes, sistema de calificaciones y seguimiento GPS en tiempo real.
⚠️ Acción requerida del cliente: Para continuar con el despliegue a producción, se requiere que el cliente defina y proporcione el servicio de hosting web (PHP + MySQL) donde se alojará el sistema.
